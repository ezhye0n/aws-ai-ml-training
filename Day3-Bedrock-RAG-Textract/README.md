## 1. Advanced Bedrock: RAG 시스템 구축
* **RAG (Retrieval-Augmented Generation):** 대형 언어 모델(LLM)의 할루시네이션(환각 현상)을 방지하기 위해, 외부 데이터 소스에서 관련 정보를 검색(Retrieval)한 후 이를 기반으로 답변을 생성하는 기술.
* **학습 내용:** Amazon Bedrock Knowledge Base와 임베딩(Embedding) 모델을 연동하여 학내 비정형 문서 데이터를 학습시키고, '학과 교육과정 Q&A 시스템'을 구현함.

## 2. Amazon Rekognition: 보호장비(PPE) 감지 및 컴플라이언스

### [2-1] 보호장비(Personal Protective Equipment) 타입 3가지
API 호출을 통해 이미지 내 인원을 감지하고 다음 3가지 필수 장비의 착용 여부 및 신체 부위(`Body Part`) 매핑 정보를 식별합니다.
* **헤드커버(Head Cover):** 안전모, 헬멧 등 머리 보호 장비
* **페이스커버(Face Cover):** 마스크, 보안경(고글) 등 얼굴 보호 장비
* **핸드커버(Hand Cover):** 안전장갑 등 손 보호 장비

### [2-2] 준수율(Compliance) 계산
* 각 인원별로 필수 장비를 모두 갖추었는지 판별하여 작업 현장의 안전 컴플라이언스 통계를 산출하고 시각화하는 실무 로직을 학습함.

## 3. Amazon Textract: 문서 AI 서비스 Core

### 💡 일반 OCR과 Textract의 차이점
* **일반 OCR:** 문자를 단순히 텍스트 줄(Line) 단위로 읽어내며, 문서 구조에 대한 이해도가 낮음.
* **Amazon Textract:** 머신러닝 기반으로 텍스트 추출을 넘어 문자의 구조적 맥락(**표, 양식, 계층 관계**)을 완벽히 분석함.

### [3-1] Textract 주요 기능 4가지
1. **Detect Document Text:** 문서 내 텍스트를 라인(Line) 및 단어(Word) 단위로 신뢰도(Confidence)와 함께 추출.
2. **Form(양식) 분석:** 문서 내의 **키-값(Key-Value) 쌍**을 구조화하여 추출.
   * *예:* `"주민등록번호: [RRN Omitted]"` $\rightarrow$ 키: `"주민등록번호"`, 값: `"[RRN Omitted]"`
3. **Table(표) 분석:** 문서 내 표 구조를 행(Row)과 열(Column)로 인식하여 셀 단위 데이터를 Pandas 데이터프레임 형식으로 즉시 변환.
4. **Query(쿼리) 기능 (★혁신 기능):** 정규식이나 위치 기반 룰(Rule) 대신, 자연어 질문을 통해 필요한 정보만 정확히 조작·추출.
   * *예:* `"주민등록번호가 뭔지 찾아줘"` $\rightarrow$ 문서 내 여러 날짜나 숫자 중 정답 데이터만 정확히 타겟팅하여 반환. 다중 오탐지 문제를 해결하는 대안.

### [3-2] 응답(Response) 구조 및 처리 방식
* **블록(Block) 구조:** API가 반환하는 기본 데이터 단위로, 텍스트 간의 부모-자식(계층) 관계를 포함함.
* **바운딩 박스(Bounding Box):** 추출된 데이터가 이미지 상의 어느 좌표에 위치하는지 기하학적 정보 제공 (시각화에 활용).
* **문서 처리 프로세스 2가지:**
  * **동기식(Synchronous):** 단일 페이지 이미지 분석 시 즉시 응답을 반환하는 방식.
  * **비동기식(Asynchronous):** 대용량 PDF(10페이지 이상) 처리 시 사용. `요청 던지기 $\rightarrow$ 백그라운드 처리 $\rightarrow$ 완료 알림(SNS/SQS) $\rightarrow$ 결과 순차 조회` 파이프라인 구조로 실무에서 주로 채택.

## 4. 문서 AI 아키텍처 설계 및 실무 고려사항

### [4-1] 입력 데이터 품질 및 포맷 제한
* 문서의 기울어짐, 저해상도, 흐림 현상은 인식 성능 저하의 주원인입니다.
* 실무 설계 시 다양한 확장자로 인한 시스템 복잡도를 낮추기 위해 **"반드시 PDF/PNG로만 제출"**과 같이 비즈니스 단에서 입력 형식을 제한하는 규칙이 중요합니다.

### [4-2] 💰 계층적 API 기반 비용 최적화 전략
대량의 문서를 처리할 때 모든 문서에 고급 API를 적용하면 비용 과다가 발생합니다.
* **1단계:** 상대적으로 저렴한 일반 텍스트 인식 API를 사용하여 문서 분류 (예: 100장의 문서 중 '진단서' 50장만 식별).
* **2단계:** 분류된 특정 문서(50장)에 대해서만 비용이 높은 고급 API(자연어 쿼리, 표 분석 등)를 선별적으로 호출하여 중복 비용 방지.

### [4-3] 인간 개입 프로세스 (Human-in-the-Loop)
* AI 서비스는 결코 100% 완벽할 수 없으므로, 금융 서류나 민감 정보 추출 시 신뢰도가 기준치 미만인 데이터는 **사람이 직접 재검토(Validation)하는 구간**을 아키텍처 설계 단계에서 명확히 정의해야 합니다.

### [4-4] ⚠️ 기술적 제약 및 한계점
* **언어 지원:** 현재 Amazon Textract는 **한글을 정식 지원하지 않으므로**, 국내 비즈니스 문서 처리 시에는 대안으로 **MS Azure** 또는 **GCP** 솔루션을 비교·검토하는 설계 시야가 필요합니다. (영문 및 글로벌 문서에 최적화).
* 특수한 도메인의 서식이나 커스텀 요구사항 처리는 기본 API 호출만으로 한계가 있어, 궁극적으로는 자체 커스텀 머신러닝 모델 개발을 고려해야 합니다.

## 5. IT 업계 현황 및 클라우드/보안 트렌드

### [5-1] 업무 문화의 변화 (2008년 vs 현재)
* **2008년:** 만연한 야근 문화, 주 6일(토요일) 근무, 타이트한 일정 관리로 인한 3D 업종 이미지 인식.
* **현재:** 주 5일제 및 워라밸(Work-Life Balance) 정착, 계획적인 업무 관리 프로세스로 개선됨.
* **AI/LLM의 영향:** 생성형 AI 도구 활용이 필수화되면서 개발자 간 기술 격차가 축소됨. 다만, 문제가 발생했을 때 근본 원인을 파악하고 해결하기 위해서는 **컴퓨터 과학 기초 지식**과 **정확한 질문(프롬프트) 설정 능력**이 더욱 중요해짐.

### [5-2] 클라우드 인프라 및 인프라 보안 현황
* **규모별 환경 선택:** 중소기업은 비용 효율적인 클라우드 집중 아키텍처를, 대기업 및 금융권은 규제 준수(금감원 감사 대비)와 개인정보 보호를 위해 **하이브리드 환경(온프레미스 + 클라우드)**을 채택함.
* **개발 환경 보안 강화:** 소스 코드 및 사내 데이터 유출 방지를 위해 로컬 PC 작업 대신 **회사에서 지급한 가상 머신(VM) 환경 내에서만 개발**하도록 통제하는 추세가 강화됨. 보안이 극대화된 일부 기업은 외부 LLM 도구 접근을 차단하기도 하여 개발자의 순수 기본기가 재강조됨.
